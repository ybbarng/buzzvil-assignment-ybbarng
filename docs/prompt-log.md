# AI 대화 프롬프트 기록

## 2026-02-28

### 1. 과제 스펙 파악

**프롬프트**: "ASSIGNMENT.md를 읽고 대략적인 과제 스펙에 대해 설명해줘."

**결과**:
- 턴제 배틀 게임: 세팅(3단계 스텝 폼) -> 전투 -> 결과 흐름
- `data-testid` 명세가 매우 구체적 — 자동화 채점에 사용될 것으로 판단
- "회사의 제품에서 실제로 사용할 프로젝트"로 구현해야 함

### 2. 기술 스택 결정

**프롬프트**: "프론트엔드 과제이긴 한데, 게임이잖아. 그러면 게임 프레임워크를 사용하는 건 어떻게 생각해?"

**결정**: 게임 프레임워크는 과하다고 판단
- 이 게임의 본질은 UI 상태 관리 (실시간 렌더링/물리 엔진 불필요)
- "프론트엔드 과제"라는 맥락에서 채점자가 보고 싶은 것은 React 역량
- `data-testid` 명세가 DOM 기반 UI를 전제

**프롬프트**: "Next.js 장점을 살릴만한 특별한 추가 기능들이 있을까? 없으면 Vite + React로."

**결정**: Vite + React 선택
- SSR, API Routes, 이미지 최적화 등 Next.js 기능이 순수 클라이언트 게임에서 활용할 곳이 없음
- Vite가 더 가볍고 빠른 개발 경험 제공

**확정 스택**:

| 영역 | 선택 | 비고 |
|---|---|---|
| 빌드 | Vite + React + TypeScript (strict) | SSR 불필요 |
| 스타일링 | Tailwind CSS v4 + shadcn-ui | cn() 유틸 사용 |
| 상태 관리 | zustand | 라우팅 없이 상태 기반 화면 전환 |
| 폼 | react-hook-form + zod | 3단계 스텝 폼 |
| 테스트 | vitest + @testing-library/react | data-testid 기반 자동 채점 대비 |
| 린트/포맷 | biome | space 2칸 |
| 패키지 매니저 | pnpm | |
| Git hooks | lefthook | pre-commit lint |

### 3. 작업 방식 설계

**프롬프트**: "이 작업은 일단 저장해두고 프로젝트 작업 방식부터 설계하고 가자"

**프롬프트**: "'회사의 제품에서 실제로 사용할 프로젝트라고 생각하고 구현해주세요' 라고 되어 있으니 다음 제약을 추가하자:
- main 브랜치에는 직접 푸시하지 마
- 모든 작업은 의미 있는 규모의 PR을 만들어서 작업해
- PR 본문은 처음 보는 사람도 이해할 수 있도록 상세하게 작성해줘
- PR을 작성하고 나면 네가 코드 리뷰어가 되어서 해당 PR을 비판적으로 검증해줘
- PR에 댓글로 리뷰를 달아줘"

**프롬프트**: "혹시 실무와 유사하게 작업하기 위해 더 필요한 부분이 있을까?"

**결정**: GitHub Actions CI와 PR 템플릿 추가
- 저장소가 PUBLIC이므로 Actions 무료 사용 가능

**프롬프트**: "CI와 PR 템플릿 구성도 해줘. 근데 첫 번째 PR에서는 언급만 하고, 이후 계획에서 구현시키는 게 좋겠지?"

**결정**: 첫 PR(docs/workflow-setup)에서는 문서에 언급만, 실제 파일은 프로젝트 초기 설정 PR에서 구현
