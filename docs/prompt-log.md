## 2026-02-28

### 1. 과제 스펙 파악

**프롬프트**: "ASSIGNMENT.md를 읽고 대략적인 과제 스펙에 대해 설명해줘."

**결과**:

- `data-testid` 명세가 매우 구체적 — 자동화 채점에 사용될 것으로 판단
- "회사의 제품에서 실제로 사용할 프로젝트"로 구현해야 함

### 2. 기술 스택 결정

**프롬프트**: "프론트엔드 과제이긴 한데, 게임이잖아. 그러면 게임 프레임워크를 사용하는 건 어떻게 생각해?"

**결정**: 게임 프레임워크는 과하다고 판단
- 이 게임의 본질은 UI 상태 관리 (실시간 렌더링/물리 엔진 불필요)
- "프론트엔드 과제"라는 맥락에서 채점자가 보고 싶은 것은 React 역량
- `data-testid` 명세가 DOM 기반 UI를 전제

**프롬프트**: "Next.js 장점을 살릴만한 특별한 추가 기능들이 있을까? 없으면 Vite + React로."

**결정**: Vite + React 선택
- SSR, API Routes, 이미지 최적화 등 Next.js 기능이 순수 클라이언트 게임에서 활용할 곳이 없음
- Vite가 더 가볍고 빠른 개발 경험 제공

**확정 스택**:

| 영역 | 선택 | 비고 |
|---|---|---|
| 빌드 | Vite + React + TypeScript (strict) | SSR 불필요 |
| 스타일링 | Tailwind CSS v4 + shadcn-ui | cn() 유틸 사용 |
| 상태 관리 | zustand | 라우팅 없이 상태 기반 화면 전환 |
| 폼 | react-hook-form + zod | 3단계 스텝 폼 |
| 테스트 | vitest + @testing-library/react | data-testid 기반 자동 채점 대비 |
| 린트/포맷 | biome | space 2칸 |
| 패키지 매니저 | pnpm | |
| Git hooks | lefthook | pre-commit lint |

### 3. 작업 방식 설계

**프롬프트**: "이 작업은 일단 저장해두고 프로젝트 작업 방식부터 설계하고 가자"

**프롬프트**: "'회사의 제품에서 실제로 사용할 프로젝트라고 생각하고 구현해주세요' 라고 되어 있으니 다음 제약을 추가하자:
- main 브랜치에는 직접 푸시하지 마
- 모든 작업은 의미 있는 규모의 PR을 만들어서 작업해
- PR 본문은 처음 보는 사람도 이해할 수 있도록 상세하게 작성해줘
- PR을 작성하고 나면 네가 코드 리뷰어가 되어서 해당 PR을 비판적으로 검증해줘
- PR에 댓글로 리뷰를 달아줘"

**프롬프트**: "혹시 실무와 유사하게 작업하기 위해 더 필요한 부분이 있을까?"

**결정**: GitHub Actions CI와 PR 템플릿 추가
- 저장소가 PUBLIC이므로 Actions 무료 사용 가능

**프롬프트**: "CI와 PR 템플릿 구성도 해줘. 근데 첫 번째 PR에서는 언급만 하고, 이후 계획에서 구현시키는 게 좋겠지?"

**결정**: 첫 PR(docs/workflow-setup)에서는 문서에 언급만, 실제 파일은 프로젝트 초기 설정 PR에서 구현

### 4. 프롬프트 기록 보존

**프롬프트**: 작성한 프롬프트를 온전하게 남기고 싶음, 전체 흐름을 알 수 있으면 좋겠다고 함

**결정**: `docs/prompt-log.md`에 날짜별 주요 프롬프트와 결정 사항 기록

## 2026-03-01

### 1. 오버워치 영웅 프리셋 — 메타데이터 기반 자동 생성

**프롬프트**: "50명 영웅 스탯을 하나하나 만들면 시간이 너무 오래 걸릴 것 같아. 역할군별 템플릿을 만들어서 자동 생성하는 방식은 어떨까?"

**결정**: 계층적 템플릿 방식 채택
- 역할군(탱크/딜러/힐러) → 서브역할군(메인탱크/오프탱크 등) → 기본 템플릿(총합 200) 정의
- 각 영웅의 오버워치 내 능력·설명에 맞게 ±3~8 미세 조정
- 원본 OW 메타데이터(`ow-hero-meta.ts`)와 변환 규칙을 분리하여, 규칙 변경 시 재생성 가능

### 2. 오버워치 스킬 변환 원칙

**프롬프트**: "프리셋 영웅을 선택하면 해당 영웅의 실제 오버워치 스킬을 기반으로 커스텀 스킬도 제공하고 싶어."

**결정**: OW 스킬의 실제 동작(description)을 기준으로 게임 스킬 타입 결정
- 수치 범위(value, mpCost)만으로 카테고리를 결정하지 않음
- 예: 겐지 "튕겨내기"는 수치상 movement에 가까워 보이지만, "투사체를 막아내는 방벽"이라는 동작에 따라 barrier로 분류
- 궁극기는 고비용·고효과, 일반 스킬은 저비용·중효과로 차등 배분

### 3. GitHub Pages 배포

**프롬프트**: "데모 사이트를 배포하고 싶어. 커스텀 도메인도 쓸 수 있으면 좋겠고."

**결정**: GitHub Pages + 커스텀 도메인(`buzz-arena.byb.kr`)
- `public/CNAME`에 도메인 기재, `vite.config.ts`의 `base: "/"`
- GitHub Actions deploy workflow 추가

## 2026-03-02

### 1. 이벤트 기반 전투 시스템 전환

**프롬프트**: "지금 전투 시스템이 단순히 상태를 바꾸는 방식인데, 이벤트 큐 기반으로 전환하면 로그 표시나 애니메이션 연출이 더 유연해질 것 같아."

**결정**: 단순 상태 변경 → 이벤트 큐(`RoundEvent[]`) 방식으로 전환
- `round-start`, `skill-use`, `skill-effect`, `defend`, `buff-expire`, `battle-end` 등 세분화된 이벤트 타입
- `pendingEvents → advanceEvent()` 타이머 메커니즘으로 순차 재생
- 기존 전투 로그를 이벤트로부터 자동 생성(`eventsToLegacyLogs`)

### 2. 인트로 애니메이션

**프롬프트**: "세팅 화면 진입할 때 게임 느낌이 좀 부족한데, 인트로 애니메이션을 추가하면 어떨까?"

**결정**: 로고 중앙 등장 → 상단 이동 → UI slide-in 시퀀스
- `IntroPhase` 3단계(`center` → `moving` → `done`)로 상태 관리
- step 전환 시에는 재생하지 않고, 페이지 로드/리마운트 시에만 재생
- `prefers-reduced-motion` 사용자는 즉시 건너뜀

### 3. 결과 화면 오버워치 스타일 개선

**프롬프트**: "결과 화면이 너무 밋밋해. 오버워치 경기 종료 화면처럼 만들 수 있을까?"

**결정**: 승리/패배/무승부별 차등 연출
- 승리: 금색 테마 + 축하 메시지
- 패배: 적색 테마 + 재도전 유도
- 무승부: 중립 테마
- 전투 분석 모달(총 턴 수, 스킬 사용 통계)

### 4. 리플레이 기능

**프롬프트**: "전투가 끝나면 다시 볼 수 없는 게 아쉬워. 리플레이 기능을 만들 수 있을까?"

**결정**: localStorage에 이벤트 저장 + 기존 전투 화면 재활용
- 전투 종료 시 전체 `RoundEvent[]`를 자동 저장 (최근 10개)
- 기존 `pendingEvents → advanceEvent()` 메커니즘을 그대로 활용하여 재생
- `isReplaying` 플래그로 일반 전투와 분기 (ActionPanel 숨김, round-start 자동 생성 차단)
- `GamePhase`에 `"replay"`, `"replay-result"` 추가하여 결과 화면에서도 분기

### 5. v2.0.0 품질 개선

**프롬프트**: "기능은 다 됐으니, 접근성이나 코드 품질 면에서 부족한 점을 점검하고 보완하자."

**결정**: 접근성·안정성·성능 세 축으로 보강
- 접근성: 시맨틱 HTML(`<main>`, `<section>`, `<header>`), ARIA 속성, 키보드 내비게이션
- 안정성: ErrorBoundary 추가, 런타임 에러 격리
- 성능: React.lazy + Suspense로 화면별 코드 스플리팅

### 3. 모바일 인트로 애니메이션 버그 수정

**프롬프트**: "모바일에서 인트로 애니메이션이 점프하는 버그가 있어. 원인을 파악하고 수정하자."

**결정**: CSS 단위 및 애니메이션 방식 변경
- `dvh` → `svh`: 모바일 브라우저 주소창 변화에 따른 뷰포트 높이 변동 제거
- CSS `@keyframes animation` → CSS `transition`: 키프레임 내 `calc()` + viewport 단위의 브라우저 해석 시점 문제 우회
- `transform: translateY(calc(50svh - 50%))` 시작 → `translateY(0)` 종착으로 transition 처리
