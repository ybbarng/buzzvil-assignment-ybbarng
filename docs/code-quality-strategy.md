# 코드 품질 개선 전략

기능 구현 완료 후 코드를 "제출 가능한 수준"에서 "실무 품질"로 끌어올리기 위해 두 단계의 반복 개선 루프를 적용했습니다.

---

## 1단계: PR 리뷰 루프

PR을 생성한 뒤 셀프 코드 리뷰를 수행하고, 발견한 문제를 개별 커밋으로 수정하는 방식입니다.

### 절차

```
PR 생성 → 셀프 리뷰 댓글 작성 → 지적 사항 커밋 → 리뷰 결과 댓글 → (반복)
```

### 리뷰 관점

6가지 관점에서 검토합니다 ([워크플로우 문서](workflow.md) 참조):

| 관점 | 예시 |
|---|---|
| 가독성 | 변수명, 함수 구조, 불필요한 복잡성 |
| 재사용성 | 중복 코드, 과도하지 않은 추상화 |
| 효율성 | 불필요한 연산·렌더링, 자료구조 |
| 보안 | 입력 검증, 민감 정보 노출 |
| 버그 가능성 | 엣지 케이스, 타입 안전성, null 처리 |
| 안정성 | 에러 대응, 테스트 커버리지 |

### 이 단계에서 발견·수정한 항목 (feature/battle-screen-ux PR)

- 전투 로그에서 이름과 조사가 줄바꿈으로 분리되는 문제 → `<span>` 래핑
- 적 턴 데미지가 음수로 표시되는 버그 → `actorAfter`/`targetAfter` 매핑 수정
- HP/MP 바 잔상 애니메이션 부재 → 감소량 잔상 추가

---

## 2단계: 평가 루프

과제 스펙과 시니어 프론트엔드 개발자 기준으로 5개 영역을 채점하고, 감점 항목을 하나씩 해소합니다.

### 채점 기준

| 영역 | 배점 | 평가 내용 |
|---|---|---|
| 기능 완성도 | 30점 | 스펙 일치, data-testid 부여, 엣지 케이스 |
| 코드 품질 | 25점 | TypeScript strict, 타입 캐스팅, 일관성, 에러 처리 |
| 테스트 | 25점 | 커버리지, 단위·통합 테스트 분포, 엣지 케이스 |
| 접근성 | 10점 | ARIA, 시맨틱 HTML, 키보드, prefers-reduced-motion |
| UI/UX | 10점 | 애니메이션, 반응성, 사용 편의, 프로덕션 대비 |

### 반복 절차

```
자체 채점 → 감점 항목 식별 → 우선순위 정렬 → 개별 수정·커밋 → 재채점 → (반복)
```

### 각 라운드별 개선 내역

#### 라운드 1 (307개 → 321개 테스트)

| 감점 항목 | 해소 방법 |
|---|---|
| stat-allocator 테스트 없음 | 6개 테스트 추가 (렌더링, 입력 변경, 랜덤 배분, 초기화) |
| step-indicator 테스트 없음 | 6개 테스트 추가 (aria-current, disabled, 클릭, trailing) |
| useCountUpProgress 훅 테스트 없음 | 2개 테스트 추가 (reduced-motion 즉시 완료) |
| result-screen 테스트 없음 | 8개 테스트 추가 (승리/패배/무승부 표시, 턴 수, 재시작, 전투 분석/다시보기 버튼) |
| 난이도 선택 aria-pressed 미검증 | className 대신 aria-pressed 속성 검증으로 교체 |

#### 라운드 2 (321개 → 335개 테스트)

| 감점 항목 | 해소 방법 |
|---|---|
| `<main>`, `<header>` 시맨틱 landmark 부재 | GameContainer의 콘텐츠 래퍼를 `<main>`으로, App·SettingScreen의 헤더를 `<header>`로 교체 |
| battle-screen 단위 테스트 없음 | 6개 테스트 추가 (라운드 표시, 패널, 스킬 버튼, 라운드 진행, 전투 종료) |

## 왜 이 전략이 효과적이었는가

1. **감점 기반 우선순위** — "이미 잘 되어 있는 것"이 아니라 "부족한 것"에 집중하여 최소 노력으로 최대 효과를 얻음
2. **개별 커밋** — 각 수정이 독립적이므로 revert 가능하고 변경 의도가 명확함
3. **자동 검증** — 매 수정마다 test/lint/build를 실행하여 회귀 방지
4. **과잉 방지** — store reset 유틸 추출, stat input aria-label 등 검토 후 over-engineering으로 판단하여 포기한 항목도 있음

---

## 포기한 개선 항목과 이유

| 항목 | 포기 이유 |
|---|---|
| store reset 유틸 추출 | 3줄짜리 코드를 유틸로 추출하면 오히려 추적이 어려워짐 |
| stat input에 aria-label 추가 | 이미 `<label htmlFor>` / `id` 연결이 되어 있어 aria-label이 기존 레이블을 덮어씀 |
| enemy-ai 매직 넘버 상수 추출 | 과제 스펙에 정의된 값이므로 인라인 유지가 적절, JSDoc 주석으로 의도 명시 |
| 전투 로그에 스킬 타입 접두사 추가 | 이미 타입별 아이콘(Sword/Heart/ArrowUp/ArrowDown)이 구현되어 있어 텍스트 접두사는 중복 |
